items系 説明書

開発者向け：「Excel列 → 出力JS → 画面での役割」整理

1. この仕組みの全体像

items系は、Excelの items シート を元にして、VBAで app/data/items/items.js を出力し、それを app/pages/items/items.html で読み込んで表示する仕組みです。

この系統は単なる一覧表示ではなく、実際には次のことをしています。

  • 元号で絞り込み
  • 会議回で絞り込み
  • 例外パターンで絞り込み
  • 番号検索 / 件名検索
  • 議案 / 請願・陳情 / 発議案の並び替え
  • meetings.js と questions.js を使った「例外判定」
💡 つまり： items系は、案件データを表示するだけでなく、会議日程と一般質問日程を参照して、その案件が通常処理か例外処理かを判定する検索画面です。

2 & 3. 出力先と基本的な考え方

元データExcelシート: items
出力先app/data/items/items.js（配列名: window.APP_DATA.items）

「1案件 = 1行」のテーブル

会議の開催行を持つ meetings系とは違い、items系は案件単位（議案1件、請願1件など）のテーブルです。ただし、各案件は meeting_id で会議回と結びついています。

  • 単体で検索対象になる
  • meeting_id を通じて meetings系と結びつく
  • questions系の一般質問日も参照して例外判定する

5. 各列の役割

5-1. item_id
役割
案件1件ごとの一意キーです。（例：TEIR0801GIJO001）
意味
一般的には meeting_id + item_class + item_subclass + item_no_numeric を組み合わせたIDです。
5-2. meeting_id 【最重要結合キー】
役割
その案件がどの会議回に属するかを示すキーです。（例：TEIR0801）
用途
meetings系との結合、items画面での会議回グループ表示、例外判定の基準日取得に使います。
5-3〜5-6. 会議回情報 (meeting_name, number_code, year, session)
meeting_name
所属する会議回の表示名。検索結果をグループ化するときの見出しとして表示。
number_code
元号コード（R, H等）。最初の元号選択で使用。
year / session
和暦年 / 回数。検索結果の並び順や、会議回選択肢の並び替えに使用。
5-7. item_class 【大分類】
役割
案件の大分類コードです。値：GI(議案), SG(請願), CJ(陳情), HG(発議案)
用途
表示ラベル生成、並び順制御、例外判定分岐。items系ではかなり重要な列です。
5-8〜5-11. 案件詳細 (item_subclass, item_no_numeric, item_no, title)
item_subclass
中分類コード（JO=条例、YS=予算等）。ID構成要素などに使用。
item_no_numeric
番号数値（3桁ゼロ埋め）。表示番号生成や並び順に使用。※VBAでゼロ埋めされます。
item_no
もとの表示用番号（例：議案第1号）。※画面側では分類コードから再構成表示もしています。
title
案件の件名。カード本文表示や件名検索に使用。
5-12. committee_id / 5-13. committee_name
役割
付託先委員会のID（SOMJ等）および表示名（総務常任委員会等）。
用途
IDは将来のロジック拡張向け。表示名はそのままカード表示されます。
5-14〜5-16. 日付群 【例外判定の中心】
teian_date
提案日。議案・発議案で「追加議案」判定に使われます。
futaku_date
付託日。請願・陳情で「通常外付託」判定に使われます。
giketsu_date
議決日。本会議最終日より前なら「先議」判定に使われます。
5-17〜5-23. その他フラグ・管理項目
result_code / label
議決結果。画面側は主に result_code を元にラベルを作っています。
syuseian_flag
1のとき、例外タグ「修正案あり」が付きます。
rireki_flag
1のとき、例外タグ「履歴あり」が付きます。
sort_order 等
並び順補助、作成日(created_at)・更新日(updated_at)、備考(note)など管理用。

6 & 7. VBA変換ルールと実例

  • 出力対象： item_id が空でない行だけを出力します。
  • 数値変換： year, session, syuseian_flag, rireki_flag, sort_order は数値として出力されます。
  • ゼロ埋め維持： item_no_numeric はゼロ埋め維持のため文字列として出力されます。
  • 正規化： 日付は yyyy-mm-dd に、フラグは 1 / 0 にVBA内で正規化されます。

ダミーデータでの並びの実例

議案（第1,2,3号） → 請願・陳情（請願1,陳情2号） → 発議案（第1,2号） の順に保持・表示されます。

8〜10. 画面側(items.html)での使われ方と並び順

⚠️ items.js 単体では動いていません
items.js, meetings.js, questions.js の3つを読み込んで動作します。

画面で行っている処理： 元号選択 → 会議回候補作成 → 検索条件絞り込み → 例外タグ計算 → 会議回ごとにカード表示

表示ラベルの生成： item_no をそのまま使うより、item_class と item_no_numeric から「議案第n号」「受理番号第n号」等と再構成しています。

並び順： 1. 年(降) → 2. 回(降) → 3. 議案/請願陳情/発議案 → 4. 番号(昇) → 5. item_id

11 & 12. 例外判定ロジックと判定条件

items系の最大の特徴です。meeting_id を使って基準日を取得し、各案件の日付と比較して例外を判定します。

取得する基準日 (meetings/questions由来)

  • plenary_first_date = 最初の本会議日
  • plenary_second_date = 2回目の本会議日
  • plenary_last_date = 最終の本会議日
  • petition_standard_date = 一般質問最終日以前の最後の本会議日

例外タグの判定条件

議案 (GI) 提案日が最初の本会議日と違う → 追加議案
議決日が最終本会議日より前 → 先議
請願・陳情 (SG, CJ) 付託日が標準付託日と違う → 通常外付託
議決日が最終本会議日より前 → 先議
発議案 (HG) 提案日が最終本会議日と違う → 追加議案
議決日が最終本会議日より前 → 先議
フラグ由来 syuseian_flag = 1 → 修正案あり
rireki_flag = 1 → 履歴あり

16. 開発上の注意点

依存ファイル必須

meetings.js と questions.js が不足すると例外表示は正しく出ません。

item_no_numeric

ゼロ埋め維持のため文字列です。比較時には数値化して使っています。

元号判定のフォールバック

画面側は number_code が無くても、年度・会議名から元号を推測する作りになっています。

出力先フォルダ

このVBAは app/data/items フォルダを自動作成しません。事前に手動作成が必要です。

17. 一文サマリ

items系は、Excelの items シートを「1案件1行」でJS化し、
meeting_id で会議回と結合しながら、meetings.js と questions.js を参照して
通常案件 / 例外案件を判定し、検索画面にカード表示する仕組み です。

  • item_id が案件の主キー、meeting_id が会議回との結合キー
  • item_class で議案 / 請願・陳情 / 発議案を判定する
  • teian_date, futaku_date, giketsu_date が例外判定の中心
  • syuseian_flag, rireki_flag も例外タグになる
  • items.html は meetings.js / questions.js も読む前提
