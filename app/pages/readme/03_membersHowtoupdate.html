members系 Excelデータベース更新マニュアル

このデータベースは、「人（議員）＋所属履歴を時系列で持つRDB構造」になっています。

■ 全体構造（超重要）

議員マスタ（人そのもの）
 ↓
各種「履歴テーブル」
 ├ 議員在任履歴（議員としての期間）
 ├ 議員_会派履歴（どの会派にいたか）
 ├ 議員_委員会所属履歴（どの委員会にいたか）
 ├ 議員_審議会所属履歴（外部委員など）
 └ 議員連絡先履歴（住所・電話など）
👉 ポイント： 「変わるものは全部履歴に逃がす」設計です。

■ ① 議員マスタ（人の基本情報）

member_id議員の一意ID （最重要）
member_no議席番号や並び順番号
member_name氏名
member_kanaふりがな（検索用）
member_name_short略称（UI用）
birth_date / age生年月日・年齢
👉 ageは計算でもOK（持たなくてもよい）
gender性別
sort_order表示順
active_flag現職かどうか（1=現職 / 0=退任）
note備考
👉 役割： 「この人は誰か」だけを持つ。
※所属や役職は絶対にここに持たない（重要）

■ ② 議員在任履歴

議員として在職していた期間を管理します。

office_term_id履歴ID
member_idどの議員か
term_start_date / term_end_date在任期間
election_label選挙種別（例: 令和5年市議選）
elected_count当選回数
end_reason_code終了理由（例: 任期満了、辞職、死亡）
👉 役割： 「議員として存在していた期間」の管理

■ ③ 会派マスタ

party_id会派ID
party_name正式名
party_name_short略称
active_flag現存するか
👉 役割： 会派の定義

■ ④ 議員_会派履歴

どの議員が、どの会派に、いつ所属していたか。

member_party_id履歴ID
member_id議員
party_id会派
role_name役職（例: 代表、幹事長）
start_date / end_date所属期間
👉 役割： 会派は変わるので履歴で持つ

■ ⑤ 委員会マスタ

committee_id委員会ID
committee_name正式名
committee_name_short略称
committee_type_code種別（例: STANDING=常任、STEERING=議運、SPECIAL=特別）
parent_event_id上位イベント（必要なら）
👉 会議との紐付け用
👉 役割： 委員会の定義

■ ⑥ 議員_委員会所属履歴

member_committee_id履歴ID
member_id議員ID
committee_id委員会ID
role_name役職（例: 委員長、副委員長、委員）
start_date / end_date所属期間
👉 役割： 委員会メンバーは毎回変わるので履歴で管理

■ ⑦ 審議会マスタ / ⑧ 議員_審議会所属履歴

⑦ 審議会マスタ

council_id審議会ID
council_name正式名
council_category_code種別（例: 附属機関、審議会）
👉 役割： 議会外の所属を管理

⑧ 議員_審議会所属履歴

※構造は「⑥ 議員_委員会所属履歴」と同じです。

■ ⑨ 議員連絡先履歴（かなり重要）

住所・電話など「変わる情報」の履歴を管理します。

contact_record_id履歴ID
member_id議員ID
start_date / end_date有効期間
連絡先情報postal_code / address / phone_home / phone_mobile / email
公開制御
（超重要）		 address_visibility
phone_home_visibility
phone_mobile_visibility
email_visibility

例: PUBLIC（公開）、SECRETARIAT（事務局まで）、STAFF（職員まで）、PRIVATE（非公開）
👉 役割： 個人情報は履歴＋公開レベル付きで管理

■ 全体の設計思想（超重要）

この設計の本質👇

  • ① マスタと履歴を完全分離
    マスタ = 定義 ／ 履歴 = 変化
  • ② 「現在」は end_date で判断
    end_date が空 → 現在進行形
  • ③ JOIN前提設計
    議員 → 会派 → 委員会 → 審議会 （全部つながる）

■ 実際の画面での使い方

  • 議員一覧
    → 議員マスタ（active_flag=1）
  • 議員詳細
    → 全履歴をJOIN（会派履歴・委員会履歴・在任履歴・連絡先履歴）
  • 委員会名簿
    → 議員_委員会所属履歴（現在のみ）
  • 会派一覧
    → 議員_会派履歴（現在のみ）

■ よくあるミス（重要）

❌ NG（悪い例）

議員マスタに「会派」「委員会」を直接入れる。

👉 履歴が壊れる・過去の情報が消える

✅ 正解（良い例）

所属や役職は、全部履歴テーブルに分けて入れる。

👉 時系列のデータが担保される
