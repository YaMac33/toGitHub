分野③・機能拡張：meeting_detail

meetings系 仕様説明書（最新版）

💡 ① システムの役割（全体像）

この機能は 👉 「会議回を軸に、すべての出来事を時系列で可視化するシステム」 です。

以下の3つを統合して表示します：

  • 会議イベント（本会議・委員会など）
  • 案件の流れ（提案・付託・議決など）
  • 一般質問（NEW）

🖥️② 画面構成（meeting_detail.html）

画面は上から順に以下の要素で構成されます。

  1. 会議回概要
  2. 月間カレンダー（メイン）
  3. 特別委員会
  4. 案件（カード表示）

📅③ カレンダー仕様（最重要）

カレンダーの最大の役割は 👉 「その日に何が起きたか」を1画面で把握する ことです。

表示されるイベント種類

  • ① 会議イベント： 本会議、議会運営委員会、常任委員会、特別委員会
  • ② 一般質問（NEW）： 通告ヘッダから生成し、同日は「一般質問（3件）」のようにまとめて表示
  • ③ 案件イベント： 提案、付託、継続審査、議決、取下げ
表示順ルール（重要）

カレンダー内は必ず以下の順で並べることで「議会の流れ」を正しく表現します。

  1. 会議
  2. 一般質問
  3. 提案
  4. 付託
  5. 継続審査
  6. 議決
  7. 取下げ

カレンダーセル表示とクリック時モーダル

  • 基本： セル内は最大3件まで表示し、それ以上は「+◯件」と略記します。
  • モーダル表示： クリックするとその日の全件詳細を表示します。
【モーダルの表示内容例】

[会議]
・本会議 / 委員会名

[一般質問]
・通告No1 山田太郎（40分）
・通告No2 佐藤花子（30分）
※noteがあれば追加（例：山田太郎（40分） / 初質問）

[案件]
・可決（議案第1号〜第5号）

🎤④ 一般質問仕様（NEW）

使用データ：通告ヘッダ（必須列）

列名役割
question_idID
meeting_id対象の会議回
question_date【新規追加】質問が行われる日付
notice_no通告番号
member_name / group_name議員名 / 会派名
allotted_minutes持ち時間
note備考

データ処理・表示ルール

  1. meeting_id で現在の会議回に絞り込み
  2. question_date でグルーピング
  3. カレンダー上は 1日1イベント（例：「一般質問（3件）」） に集約して件数表示
  4. モーダル内で「通告No、議員名、会派名、持ち時間、note」を展開表示

🗂️⑤ 案件表示（カードUI）

カレンダーの下部に、案件の流れをパターンごとにカード表示します。

表示順（確定）と各カードの内容

  1. 基本形（議案）： 提案日、付託日、議決日を表示
  2. 請願・陳情： 付託日（＝上程日扱い）、議決日を表示
  3. 発議案： 提案日＝議決日 として表示
  4. 例外（あれば）： 先議、追加議案、追加議案（先議）、継続審査、取下げ
【基本形（議案）の表示例】
議案第1号〜第5号
  提案日：◯年◯月◯日
  付託日：◯年◯月◯日
  議決日：◯年◯月◯日

【例外カードの表示例】
追加議案（先議）
議案第10号
  提案日：◯日
  議決日：◯日

👉 非表示ルール： 例外が0件の場合は、例外カード自体を表示しません。

🔘⑥ カードUI仕様（開閉アクション）

  • 初期状態： 高さを固定し、一部の案件のみを表示します。
  • ボタン動作： 「もっと見る」を押すと全件表示に展開し、ボタンの文字が「閉じる」に変わります。
  • 対象カード： 基本形、請願・陳情、例外すべてに適用されます。

⚙️⑦ 案件判定ロジック（超重要）

案件がどのパターンのカードに分類されるかは、以下の流れで判定します。

区分判定ロジック（アクションの流れ）
基本形（議案）PROPOSED → REFERRED → DECIDED
発議案PROPOSED → DECIDED （同日）
請願・陳情PROPOSED（＝付託） → DECIDED
継続審査（決算）一般質問最終日：提案
最終日：付託 ＋ 継続審査
継続審査（通常）最終日：継続審査
例外：先議提案日 ＝ 議決日 （付託なし）
例外：追加議案提案日 ≠ 初日
例外：追加（先議）追加 ＋ 先議

🔄⑧ カレンダー統合のポイント

会議、案件、一般質問という出所の異なるデータを、すべて同じ形式（イベントオブジェクト）に変換してカレンダーに流し込みます。

  • 変換フォーマット： date, label, type, sort_order
  • type 一覧： meeting, question, item
  • 表示制御： 同日イベントはまとめ、同種イベントは件数などで集約表示します。

🗄️⑨ データ設計まとめ

この画面を動かすための必須テーブル（シート）です。

  • meetings（会議回）
  • events（会議イベント）
  • items（案件）
  • item_actions（案件履歴）
  • questions（通告ヘッダ・一般質問）

🛠️⑩ 保守ポイント（非エンジニア向け）

修正する場所の目安
  • 一般質問を追加したい： 👉 Excelの通告ヘッダ（questions）に1行追加
  • カレンダーがおかしい： 👉 まず question_date, action_date, meeting_id を確認
  • 案件の分類がおかしい： 👉 action_type を確認
  • 表示される文字が変： 👉 result_label を確認

⚠️⑪ よくあるミス（エラーの原因）

❌ question_date が未入力カレンダーに出現しません。
❌ meeting_id が違う現在の会議回の画面に表示されません。
❌ 日付の形式が違うカレンダーの並びが崩れたり、エラーになります。

✨⑫ この設計の強み

  • 会議単位で「すべて」が見える
  • Excelの運用フローをそのまま使える
  • 「追加」や「先議」などの例外パターンにも対応できる
  • UIが崩れにくい堅牢なカードとカレンダーの設計
  • サーバー不要でLGWAN環境でも軽快に動く（完全静的）
