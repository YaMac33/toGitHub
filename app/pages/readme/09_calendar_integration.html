💻第1部：開発者向け説明書（設計・実装仕様）

1. 概要

本機能は meeting_detail 画面において、以下の2つの要素を月間カレンダーに統合表示します。

  • 会議イベント（本会議・委員会等）
  • 案件進行（提案・付託・議決等）

また、下部の「案件」欄では、進行パターン単位での要約表示を行います。

2. データ構造

2-1. 入力データ

以下のデータマスタおよび履歴を使用します。

  • meetings
  • events
  • items
  • item_actions
  • special_committee_*

2-2. 案件単位データ（内部生成）

itemUnit {
  item_id
  item_class
  item_no
  item_no_numeric
  title

  proposed_date
  referred_date
  decided_date
  continued_date
  withdrawn_date

  decided_result_label
}

2-3. 統合イベント形式

calendarEvent {
  date
  category   // meeting | item
  label      // 表示文字
  sort_order
  css_class
  detail_text
}

3. 処理フロー

  1. ① 案件アクション取得
    item_actions WHERE meeting_id = ? を取得します。
  2. ② 案件単位へ集約
    action_type ごとに日付を割り当てます。
    action_type格納先
    PROPOSEDproposed_date
    REFERREDreferred_date
    DECIDEDdecided_date
    CONTINUEDcontinued_date
    WITHDRAWNwithdrawn_date
  3. ③ 標準提案日算出
    対象：item_class = BILL AND action_type = PROPOSED
    ロジック：最頻出日を採用します。
  4. ④ 区分判定
    以下の判定順（上から優先）で区分を出力します。
    1. withdrawn
    2. proposal
    3. petition_request
    4. additional_fasttrack
    5. additional_continued
    6. additional
    7. fasttrack
    8. basic_bill
    9. other
  5. ⑤ グルーピング
    同一条件（区分、日付構成、result_label）でまとめます。
  6. ⑥ カレンダーイベント生成
    各案件からカレンダー表示用のデータを生成します。
    項目表示内容
    proposed_date提案
    referred_date付託
    decided_dateresult_label の値
    continued_date継続審査
    withdrawn_date取下げ
  7. ⑦ 会議イベントと統合
    meetingEvents + itemEvents を結合してカレンダーに描画します。

4. カレンダー表示仕様

4-1. 並び順

  1. 会議
  2. 提案
  3. 付託
  4. 継続審査
  5. 議決系
  6. 取下げ

4-2. 表示制限

  • 最大表示件数：3件まで
  • 超過した場合：+N件 のように略記表示

4-3. 表示例

3/25
・本会議
・可決（議案第1号〜第5号）
・継続審査（議案第6号）
+2件

4-4. モーダル

  • 日付や「+N件」をクリックすると、全件を同一の並び順で表示するモーダルを展開します。

5. 案件欄表示仕様

5-1. 大枠の表示順

  1. 基本形（議案）
  2. 請願・陳情
  3. 発議案
  4. 例外

5-2. 「例外」内の順序

  1. 先議
  2. 追加議案
  3. 追加議案（先議）
  4. 追加議案（継続審査）
  5. 取下げ
  6. その他

5-3. 画面表示例

【基本形（議案）】
議案第1号〜第5号:
  提案日: R6/2/20
  付託日: R6/2/21
  議決日: R6/3/25

【請願・陳情】
受理番号第1号:
  付託日: R6/2/21
  採択日: R6/3/25

6. 重要設計ルール

① カレンダーと案件欄の役割分離
項目役割
カレンダー日付ベース（その日に何が起きたか）
案件欄パターンベース（案件がどういう流れで進んだか）
  • ② result / result_label の役割：
    result はシステム上の判定用、result_label は画面表示用として厳格に使い分けます。
  • ③ 請願・陳情の扱い：
    上程日は表示せず、付託日＝開始扱いとします。
  • ④ 発議案の扱い：
    提案日と議決日は同日可とします。

📝第2部：非エンジニア向け保守説明書

1. この画面で何ができるか

この画面では、以下の2つの情報を直感的に確認できます。

  • 会議の日程
  • 議案・請願などの処理状況

これらを「カレンダー（日別）」と「一覧（案件別）」の両方から確認することができます。

2. 見方

カレンダーの見方

1日ごとに「会議（本会議など）」と「議案の動き（提案・付託・可決など）」が表示されます。

【例：3月25日のマス】
・本会議
・可決（議案第1号〜第5号）
・継続審査（議案第6号）

※日付や「+2件」などの文字をクリックすると、その日のすべての詳細が表示されます。

3. 下の「案件」欄

案件の流れをパターンごとにまとめて表示します。見出しには以下の意味があります。

見出し意味
基本形通常の議案の流れ
請願・陳情市民から提出されたもの
発議案議員から提出されたもの
例外先議、追加議案、取下げなどの特殊ケース

4. データ修正方法

修正を行う場合は、Excelの以下のシートを編集します。

  • 案件マスタ (items)
  • 案件アクション履歴 (item_actions)
  • 任期マスタ (council_terms)

よくある修正ケース

  1. 可決・否決の修正： result_label を変更します。（例：可決 → 否決）
  2. 日付の修正： action_date を正しい日付に修正します。
  3. 継続審査になった場合： action_type を CONTINUED にし、result_label を「継続審査」にします。
  4. 取下げになった場合： action_type を WITHDRAWN にします。

5. 入力時の注意点（NGと正しい例）

❌ よくあるNG（エラーの原因）
・日付が未入力（空欄）になっている
・item_id が案件マスタとアクション履歴で一致していない
・action_type の英語の綴りが間違っている
✅ 正しい入力例（議決の場合）
action_type: DECIDED
result: PASSED
result_label: 可決

6. 更新手順

データを修正して画面に反映させるまでの流れです。

① Excel修正 ➔ ② VBA実行 ➔ ③ JS生成 ➔ ④ GitHub反映 ➔ ⑤ 画面確認

7. よくあるトラブル

現象原因・確認するところ
案件が表示されないマスタと履歴で item_id が一致していないか確認してください。
日付がおかしいaction_date の入力形式（YYYY-MM-DDなど）が正しいか確認してください。
カレンダーに出ないmeeting_id が対象の会議回のものと一致しているか確認してください。
🔥 最後に（保守の要点）

この画面と機能拡張における設計のポイントは以下の2点です。

  • カレンダー ＝ 日付の事実（いつ何が起きたか）
  • 案件欄 ＝ 流れの整理（その案件がどう進んだか）

この役割分担の原則を崩さない限り、今後どのような特殊な案件パターンが増えたとしても、システムが破綻することなく拡張し続けることができます。
