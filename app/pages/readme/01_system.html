システム概要・開発者向けガイド

本ドキュメントは、システム管理者や将来の開発・保守担当者に向けて、本システムのアーキテクチャ、ディレクトリ構造、および拡張方針について解説します。

コア設計思想

一言で表すと：
「Excelをデータベース（正本）として、ブラウザ（JS）でJOINして表示するシステム」

  • Excel = DB（正本）
  • JS = 配信されるデータ（APIの代替）
  • HTML/CSS = ビュー（フロントエンド）

1. アーキテクチャの特徴

  • サーバ不要（完全静的ホスティング）： Webサーバ側でのデータベース構築や動的処理（PHP, Ruby, Node.js等）が不要です。LGWAN環境のローカルネットワークや、静的ファイルホスティング環境でそのまま動作します。
  • Excelベースの運用： 非エンジニアの職員でも、使い慣れたExcelインターフェースで安全にデータの追加・修正が可能です。
  • 履歴ベースの設計： データを上書きせず、期間（開始日〜終了日）を持たせた履歴テーブルとして保持することで、過去の任意の時点の状態を再現可能なRDB的構造を持っています。

2. ディレクトリ構成

システム全体は以下の構造で配置されています。

project/
├─ source.xlsm            ← 正本データ（Excel・VBA内包）
│
└─ app/                   ← Webアプリのルート
   ├─ index.html          ← アプリの入り口
   │
   ├─ pages/              ← 画面（HTML）群
   │  ├─ members/        ← 議員・詳細画面
   │  ├─ meetings/       ← 会議回・イベント画面
   │  ├─ items/          ← 案件管理画面
   │  ├─ questions/      ← 通告検索システム
   │  └─ readme/         ← 本ドキュメント群
   │
   ├─ js/                 ← ロジック群
   │  ├─ core/           ← ユーティリティ、共通フォーマッタ
   │  └─ (各ドメイン)/   ← 画面ごとの描画制御JS

      ├─ committees/     ← committees.js など
      └─ ...

3. データの紐づけ原則（リレーション）

すべてのデータは名前ではなくIDをキーとしてJOINされます。フロントエンドのJS側で、このIDを元に関連データを配列から抽出・結合して画面を描画します。

  • member_id → すべての個人履歴（所属、連絡先など）の中核
  • party_id → 会派情報
  • committee_id → 委員会情報
  • meeting_id → 会議回ごとのスケジュールやイベント

4. VBAによるデータ配信の仕組み

ExcelのVBAマクロは、各シートの内容をJSON形式の配列として構築し、それをJavaScriptの変数（window.APP_DATA.〜）に代入する形の .js ファイルとして書き出します。

エクスポート手順

  1. Excel上で Alt + F8 を押下。
  2. 初回または全体更新時： ExportAllData を実行（全シートを一括出力）。
  3. 一部更新時： 修正したシートをアクティブにし、ExportActiveSheetData を実行。
出力先は app/data/ 以下の各ドメインフォルダです。出力されたJSファイルは、そのままブラウザから読み込まれ、データベースの役割を果たします。

5. 開発方針と今後の拡張性

本システムは「同じ設計パターン」を踏襲することで、容易に機能を拡張できます。

基本ステップ

  1. まず動かす： 新しいExcelシートを作成し、データをIDで紐付ける。
  2. データ分離： app.js などの共通ロジックに依存しすぎず、ドメインごとにJSとビュー（HTML）を分離する。
  3. UI強化： データが正しくJOINできることを確認した後に、見た目を整える。

拡張のアイデア

以下のような機能も、現在のディレクトリ構成（新しいフォルダとHTML/JSを追加）とExcelシートの追加のみで実装可能です。

  • 表決履歴（どの議員がどの議案に賛成・反対したか）
  • 行政視察履歴（いつ・どこへ・何の目的で視察したか）
  • 政務活動費の支出管理
