議員管理システム（内部向けREADME）

1. 概要

本システムは、市議会議員の情報をExcelで管理し、Webアプリとして参照する仕組みです。

Excel（正本データ）
↓
VBAでJS生成
↓
HTML + JavaScriptで表示（LG環境対応）

特徴

  • サーバ不要（LGWAN環境対応）
  • Excelが正本（非エンジニアでも運用可能）
  • 履歴ベース設計（在任・会派・委員会など）
  • IDでデータを紐づけ（DB的構造）

3. データ設計

基本ルール

  • 1シート = 1JSファイル
  • 1行 = 1レコード（オブジェクト）
  • 1行目 = フィールド名（キー）
  • IDでデータを紐づける
  • 履歴は「開始日～終了日」で管理

主なテーブル

議員マスタ
members.js

  • 基本情報（氏名、生年月日など）

在任履歴
office_terms.js

  • 任期単位で管理
  • 現職/過去判定に使用

会派
parties.js / member_parties.js

  • 会派マスタ + 所属履歴

委員会
committees.js / member_committees.js

審議会
councils.js / member_councils.js

連絡先
contacts.js

  • 住所・電話・メール（履歴管理あり）

4. データの紐づけ（重要）

すべてIDで紐づける

  • member_id → 全履歴
  • party_id → 会派
  • committee_id → 委員会
  • council_id → 審議会

例：

members
  ↓ member_id
member_parties
  ↓ party_id
parties

5. 「現在」の考え方

以下を満たすものを「現在」とする

start_date <= 今日
AND
(end_date が空 or end_date >= 今日)

必須ルール

  • IDは絶対に変更しない
  • 名前で紐づけない

よくあるミス

  • M001 と M1 の混在
  • 全角スペース混入
  • シート名ズレ
  • ヘッダ空欄

9. トラブル対応

JSが表示されない
→ dataフォルダ未生成 or VBA未実行

特定データだけおかしい
→ ID不整合

現在判定がおかしい
→ 日付のズレ

14. meetings.html（議会回・委員会ビュー）について

meetings.html は、議員個人ではなく、会議回（定例会・臨時会）を起点に議会日程や特別委員会を確認するための画面です。

14-1. 画面の目的

  • どの会議回で、どのイベントがあったかを確認する
  • どの会議回で、どの特別委員会が設置されたかを確認する
  • その特別委員会の委員名簿を確認する
  • 会期外開催でも、当該会議回に紐づく議会運営委員会や会派代表者会議を確認する

14-2. 読み込むデータ

ファイル 役割
meetings.js会議回マスタ（第○回定例会・臨時会の枠）
event_types.jsイベント種別マスタ（本会議、常任委員会、議会運営委員会、会派代表者会議 など）
events.js議会イベント履歴
special_committees.js特別委員会マスタ
special_committee_instances.js特別委員会設置履歴
special_committee_meetings.js特別委員会会議日
special_committee_members.js特別委員会委員名簿履歴
committees.js常任委員会・議会運営委員会などの委員会マスタ
members.js委員名簿表示用の議員マスタ

14-3. データの考え方

meetings.html では、次の考え方でデータを扱います。

  • 会議回 = 枠
  • イベント = その会議回に関連する出来事
  • 特別委員会 = 設置される対象
  • 委員名簿 = その特別委員会に所属した議員

重要なのは、イベント日が会期外でも、当該定例会・臨時会のためのイベントであれば、その会議回に紐づけることです。

14-4. 会期外イベントの扱い

議会運営委員会や会派代表者会議は、会期開始前に行われることがあります。
その場合でも、どの会議回のためのイベントかが明確であれば、meeting_id はその会議回を設定します。

項目 考え方
meeting_idそのイベントが対象とする会議回
event_date実際の開催日

例：

第1回定例会の開始日: 2024-02-20
議会運営委員会の開催日: 2024-02-13

→ 会期外開催だが、meeting_id は 第1回定例会 を設定する

14-5. meetings.html で確認できること

  • 会議回一覧
  • 会議回ごとの本会議日程
  • 会議回ごとの議会運営委員会開催日
  • 会議回ごとの会派代表者会議開催日
  • 会議回ごとの常任委員会開催日
  • 会議回ごとの特別委員会一覧
  • 特別委員会ごとの会議日
  • 特別委員会ごとの委員名簿

14-6. 会議回マスタに追加した列

会議回の詳細表示では、将来的に日程表へのリンクも表示する想定です。
そのため、会議回マスタに次の列を追加しています。

列名 内容
schedule_file_path日程表ファイルへのパス

14-7. 特別委員会設置履歴に追加した列

特別委員会の詳細表示では、将来的に名簿ファイルへのリンクも表示する想定です。
そのため、特別委員会設置履歴に次の列を追加しています。

14-8. 今後の拡張方針

  • 会期日数の表示
  • 議会運営委員会開催日の集約表示

15. 案件管理ページ（議案・請願・陳情・発議案）

議会で扱われる「議案・請願・陳情・発議案」を一元管理し、
「どの会議で何が提案・処理されたか」を双方向で追跡できるようにしたページです。

15-1. 管理対象

  • 議案（条例・予算・決算・契約・人事・財産・専決処分など）
  • 発議案（条例・意見書・決議など）
  • 請願
  • 陳情

15-2. 案件マスタの役割

案件そのもの（タイトル・分類・番号など）を管理します。

項目 内容
item_id内部用ID（変更不可）
item_no表示用番号（議案第○号など）
item_class大分類（議案・請願・陳情・発議案）
item_subclass中分類（条例・予算など）
numbering_scope番号体系（YEAR / TERM）
term_id任期単位番号用（請願・陳情のみ）
year / fiscal_year年・年度（西暦）
year_wareki / fiscal_year_wareki和暦表示用
title件名
summary概要（検索用）
is_budget予算系フラグ（補正予算抽出用）

15-3. 番号体系の違い

  • 議案・発議案 → 年ごとに番号付与
  • 請願・陳情 → 任期ごとに番号付与

この違いは numbering_scope と term_id によって管理します。

15-4. 案件アクション履歴の役割

「その案件が会議の中でどう処理されたか」を記録します。

項目 内容
item_id対象案件
meeting_idどの会議回か
event_idどのイベントか
action_type処理内容（提案・付託など）
result結果（通った / 通らなかった）
action_date処理日

15-5. action_type と result の考え方

処理内容と結果は分離して管理します。

  • action_type → 何が起きたか（提案・付託など）
  • result → 最終結果（通った / 通らなかった）

なお、最終結果は DECIDED のときのみ記録します。

15-6. 結果の統一ルール

  • 可決・同意・承認・認定・採択 → 「通った（PASSED）」
  • 否決・不採択 → 「通らなかった（REJECTED）」

細かい用語の違いは note に記録します。

15-7. この設計でできること

  • 「この定例会で何が提案・議決されたか」の一覧表示
  • 「議案第○号がどの会議でどうなったか」の追跡
  • 補正予算のみの抽出
  • 請願・陳情の任期単位管理

15-8. 今後の拡張方針

  • 議案ごとの詳細ページ（PDFリンクなど）
  • 委員会審査の詳細表示
  • 議案検索機能の強化（全文検索）

まとめ

  • サーバ不要の議員管理システム
  • Excelベースで誰でも更新可能
  • 履歴管理 + ID結合で柔軟性あり
