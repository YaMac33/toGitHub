items系 Excelデータベース更新マニュアル

このデータベースは、「案件そのもの」と「案件が会議の中でどう動いたか」を分けて管理する設計になっています。

任期マスタ
↓
案件マスタ
↓
案件アクション履歴

1. 任期マスタ

議会の任期そのものを持つマスタです。（議員個人の在任履歴ではなく、議会全体の任期です）

各列の定義

列名 説明 例
term_id 任期IDです。主キーです。 TERM-R5
term_name 表示用の任期名です。 令和5年改選任期
start_date 任期開始日です。 2023-05-01
end_date 任期終了日です。 2027-04-30
note 補足メモです。特記事項があるときだけ使います。
created_at / updated_at 作成日・更新日です。（データ管理用）
役割：
案件番号が「任期ごとに振り直し」のものを扱うときに重要です。特に請願・陳情のような「受理番号第○号」を扱うときに効きます。

2. 案件マスタ

案件そのものの基本情報を持つテーブルです。「何という案件か」「何分類か」「何号か」を持ちます。

各列の定義

列名 説明 例 / 補足
item_id 案件IDです。主キーです。（内部管理用の一意ID） BILL-2024-001
item_class 大分類です。 BILL = 議案
PROPOSAL = 発議案
PETITION = 請願
REQUEST = 陳情
item_subclass 中分類です。 ORDINANCE = 条例
BUDGET = 予算
CONTRACT = 契約
RESOLUTION = 決議
item_no 表示用の番号です。 議案第1号、発議案第2号、受理番号第1号など
item_no_numeric 番号の数値部分だけです。（並び替えや検索用） 1, 2
numbering_scope その番号が何単位で振り直されるかです。
「議案は年単位」「請願は任期単位」等のルールを表現します。		 YEAR = 年ごと
TERM = 任期ごと
term_id 任期マスタへの参照です。任期単位管理の案件で使用。（年単位の案件では空欄可）
fiscal_year 年度の西暦値です。 2024
fiscal_year_wareki 年度の和暦表示です。 R6
year 年の西暦値です。 2024
year_wareki 年の和暦表示です。 R6
title 案件名です。 〇〇条例の制定について
summary 案件概要です。（検索や詳細表示用）
department 所管課です。 総務課
is_budget 予算系案件かどうかです。（補正予算の抽出などに便利） 1 = 予算系
0 = 予算系ではない
note 補足メモです。承認・認定・採択送付など、分類に入れきらない情報を置けます。
sort_order 表示順です。（必要に応じて使用）
created_at / updated_at 作成日・更新日です。

💡 fiscal_year と year の違い（重要）

fiscal_year = 年度 ／ year = 年
たとえば、2025年1月の案件でも、年度としては2024年度のことがあります。そのため、両方持っておくと後で困りません。

役割：
案件マスタは、あくまで「この案件は何か」を持つ場所です。
まだここには「いつ提案されたか」「いつ通ったか」は入れません。

3. 案件アクション履歴

案件が会議の中でどう扱われたかを持つ履歴です。本設計の肝となるテーブルです。

各列の定義

列名 説明 例 / 補足
action_id 案件アクション履歴IDです。主キーです。
item_id どの案件かを示すIDです。案件マスタの item_id を参照します。
meeting_id どの会議回で扱われたかです。会議回マスタにつながります。
event_id どのイベントで扱われたかです。（本会議、議運、委員会などの単位）
action_type その会議で何をしたかです。 PROPOSED = 提案
REFERRED = 付託
REPORTED = 報告
CONTINUED = 継続審査
WITHDRAWN = 取下げ
DECIDED = 最終結果
result そのアクションの結果です。
特に DECIDED のときに重要です。
※画面上は item_class に応じて「可決」や「採択」と表示を変えると分かりやすいです。		 NONE
PASSED（議案なら可決、請願なら採択）
REJECTED
action_date そのアクションがあった日です。
note 補足メモです。ここで細かいニュアンスを持ちます。 承認、認定、同意、採択送付 など
created_at / updated_at 作成日・更新日です。
役割：
案件アクション履歴は「案件の時系列」を持つ場所です。
たとえば1つの案件について、2/20 提案 → 2/21 委員会へ付託 → 3/10 委員会報告 → 3/25 可決 のような流れを全部ここに入れます。

4. この3つの関係

任期マスタ
  1 ── n 案件マスタ

案件マスタ
  1 ── n 案件アクション履歴
  • 1つの任期に複数の案件が属する
  • 1つの案件に複数の履歴がぶら下がる

5. なぜ分けるのか

これを1つの表に全部入れると、同じ案件名が何回も出てきて管理しづらくなります。

【悪い例（フラットな表）】
議案第1号 〇〇条例 2/20 提案
議案第1号 〇〇条例 2/21 付託
議案第1号 〇〇条例 3/25 可決

これだと案件名や分類が毎回重複します。だから、以下のように分けます。

  • 案件そのもの は 案件マスタへ
  • 動き は 案件アクション履歴へ

これがリレーショナルデータベース（RDB）として自然な設計です。

6. 実務上の見方

  • 案件管理ページ
    案件マスタ を主に使う → 一覧表示向き
  • 案件詳細ページ
    案件マスタ ＋ 案件アクション履歴 を使う → 履歴表示向き
  • 会議詳細ページ
    meeting_id で 案件アクション履歴 を引く → その会議で扱った案件一覧が出せる

7. 特に重要な列

設計上、運用においてかなり大事になる列は以下の通りです。

  • numbering_scope : 番号体系を年単位か任期単位かで分けるため
  • term_id : 任期ごとの番号管理に必要
  • fiscal_year / year : 年度と年を両方持つため
  • action_type : 履歴の意味を持たせるため
  • result : 最終結果を共通コードで持つため
  • note : 細かいニュアンスを逃がすため
