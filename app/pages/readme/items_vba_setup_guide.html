items.js 生成VBA 作成手順説明書

この説明書は、items シートを元に app/data/items/items.js を生成する VBA を作る前段階として、 現時点で決まっていること、まだ決める必要があること、作成までの順番を詳しく整理したものです。

この説明書の対象

対象は、案件データの Excel 正本から items.js を吐き出す仕組みです。

現時点での前提は次のとおりです。

項目現時点の前提
シート名items
出力対象app/data/items/items.js
用途Webアプリで案件一覧・詳細表示に使う
設計方針代表値は案件マスタに持ち、途中経過や例外は履歴テーブルで補う
表示方針該当しない列があってもアプリで表示できなくてよい

1. いま決まっていること

1-1. シート名は items にする。

1-2. 列名は、直前に確定した items 用の列名を使う。

1-3. 該当しない列があっても、アプリで表示しなくてよい。

1-4. 案件マスタには「一覧で見たい代表値」を持たせる。

2. この設計での考え方

2-1. items シートは「案件の代表値」を持つ

proposed_date、 first_referred_date、 final_vote_date、 final_result など、 一覧で見たい情報を 1 行にまとめて持たせます。

2-2. 例外的な経過は履歴テーブルで持つ

再付託、継続審査、差戻し、複数回の議決など、 1 案件 1 行に収まりにくいものは別の履歴テーブルで管理する想定です。

2-3. Webアプリは items.js の全部を表示しなくてもよい

Excel では管理用として列を持っていても、アプリ側で全部使う必要はありません。 まずは正本として必要な情報を確保し、画面表示はあとで必要な分だけ使えば大丈夫です。

3. VBA 作成前に最低限確認が必要なこと

ここが未確定だと、VBA は書けても後で修正が出やすくなります。 先にこの項目を決めておくと、作り直しがかなり減ります。

3-1. item_id の作り方

最重要です。item_id をどのルールで作るかを確定します。

確認したいこと例
手入力にするかExcelで item_id を直接入力する
自動生成にするかmeeting_id + item_class + item_no_numeric など
同一案件の一意性を何で担保するか回＋分類＋番号 など

3-2. 出力元は items シートそのものか

いまの理解では items シートをそのまま出力元にする前提です。 もし「入力用シート」から転記して items を作るなら、VBAの構成が変わります。

3-3. 出力先パス

VBA でどこに書き出すかを決めます。たとえば次のような構成です。 

project/
├─ source/
│  └─ source_items.xlsm
└─ app/
   └─ data/
      └─ items/
         └─ items.js

この場合、Excelファイルの配置場所と出力先の相対位置を合わせておく必要があります。

3-4. 日付の出力形式

たとえば 2026-04-03 のように統一するのか、 空欄は "" にするのかを決めます。

3-5. フラグ列の値

is_budget、 has_amendment_flag、 has_history_flag を 0/1 にするのか、 TRUE/FALSE にするのかを決めます。

3-6. 空行・未入力行の扱い

どの条件で出力対象から除外するかを決めます。 実務では、item_id が空の行は出力しない、のようなルールが一般的です。

4. 実際に作成するまでのおすすめ手順

手順1：items シートを完成させる

まずは Excel の items シートに、確定した列名を 1 行目に並べます。 並び順もこの段階で固定するのがおすすめです。

理由は、VBA 側で「見出し名で列を探す」方式にもできますが、最初に並びを固めた方が管理しやすいからです。

手順2：1〜3行だけダミーデータを入れる

実データを大量に入れる前に、少数のサンプル行で確認します。

確認したいこと内容
日付の入力例空欄・通常の日付・特殊な日付を試す
フラグの入力例0 と 1 を試す
文字列の入力例件名、概要、メモに長文や改行を含める

手順3：item_id ルールを確定する

後から item_id のルールを変えると、既存データや画面側の紐付けも影響を受けやすくなります。 ここは VBA 作成前に固めるのが安全です。

手順4：ファイル配置を決める

Excel と app フォルダの位置関係を決めます。 たとえば、Excel は source フォルダ、 出力先は app/data/items にすると整理しやすいです。

手順5：出力形式を固定する

JS は通常、次の形式を想定します。

window.APP_DATA = window.APP_DATA || {};
window.APP_DATA.items = [
  {
    item_id: "....",
    title: "...."
  }
];

この形式でよければ、その前提で VBA を作れます。

手順6：VBA を作成する

上の条件が決まったら、Excel から rows を順に読み、 JS 文字列として整形して items.js に書き出す VBA を作成します。

手順7：Webアプリ側で読み込み確認する

生成した items.js をブラウザで読み込み、 エラーが出ないかを確認します。

5. items シート作成時のおすすめルール

5-1. 見出し名は完全一致にする

VBA 側で見出し名をもとに列を取得するなら、 大文字・小文字やアンダースコアを含めて統一しておくのが安全です。

5-2. 日付列は日付として入力する

文字列でばらつくよりも、Excel の日付セルとして入力した方が、 YYYY-MM-DD 形式への変換が安定します。

5-3. フラグ列は 0 / 1 に統一する

Web側で扱いやすく、Excelでもわかりやすいので、 TRUE/FALSE より 0/1 の方が実務では安定しやすいです。

5-4. 空欄は空欄のまま持つ

該当しない情報まで無理に埋めず、空欄にしておく方があとで扱いやすいです。

6. いまの段階で「これだけ決めれば VBA を書き始めやすい」項目

優先度項目理由
最優先item_id の生成ルール一意性の基準になるため
最優先Excelファイルの配置場所相対パス計算に必要なため
高items シートの列順VBA の安定性が上がるため
高出力先ファイル名書き出し処理で固定するため
中フラグ値の形式後から直す手間を減らすため
中空行の判定ルール不要行を出力しないため

7. 現時点でおすすめするファイル名と配置

この説明書自体は、次のような名前で保存しておくのがおすすめです。

project/app/pages/readme/items_vba_setup_guide.html

Excel 正本の例：

project/source/source_items.xlsm

JS 出力先の例：

project/app/data/items/items.js

まだ最終構成が固まっていない場合でも、上のように役割ごとに分けておくと後で混乱しにくいです。
