分野⑥：議題管理

議題（アジェンダ）管理

このページでは、議案等とは別に「議運や会派代表者会議などで、どんな議題が、いつ、どの会議で扱われたか」を追えるようにするための設計と運用仕様を解説します。

💡 基本概念：この機能でできること
  • 会議回詳細ページで、その回に扱われた議題を確認する
  • 議題一覧ページで、議題ごとの履歴件数や関係会議数を確認する
  • 議題詳細ページで、1つの議題がいつどの会議で扱われたかを時系列で追う

📁1. 関係ファイル

1-1. ページ（HTML）

app/pages/agendas/agenda.html

議題一覧ページ。議題ごとに検索し、詳細ページへ遷移します。

app/pages/agendas/agenda_detail.html

議題詳細ページ。議題ベースで履歴を確認します。

app/pages/meetings/meeting_detail.html

会議回詳細ページ。会議回ベースで議題を確認します。

1-2. ロジック（JS）

app/js/agendas/view_agendas.js

議題一覧ページの表示ロジックです。

app/js/agendas/view_agenda_detail.js

議題詳細ページの表示ロジックです。

app/js/meetings/view_meeting_detail.js

会議回詳細ページの中で、議題も表示するロジックを持っています。

1-3. データソース

app/data/agendas/agendas.js

議題マスタです。

app/data/agendas/agenda_actions.js

議題アクション履歴です。

app/data/meetings/events.js

会議イベントマスタです。議題履歴の表示順や会議種別判定に使います。

💾2. データベース設計

2-1. 議題マスタ（agendas シート）

議題そのものを管理するマスタです。議題名や分類を持ちます。

列名（内部キー）意味
agenda_id議題ID
agenda_category大分類（議運、会派代表者会議、その他）
agenda_subcategory中分類（日程調整、議案取扱、報告など）
agenda_title議題名
agenda_summary概要
note備考
sort_order表示順

2-2. 議題アクション履歴（agenda_actions シート）

議題がどの会議で、いつ、どのように扱われたかを持つ履歴です。議題を追う中心テーブルになります。

列名（内部キー）意味
agenda_action_id議題履歴ID
agenda_id議題ID
meeting_id会議回ID
event_idイベントID
action_type扱い（議題提示・協議・決定・報告・継続など）
action_date扱った日
note備考

2-3. events の役割

agenda_actions だけでは「同じ日に複数の議運がある場合の順番」を持てないため、events 側の情報も併せて使います。

列名（内部キー）意味
event_id会議イベントID
event_date開催日
event_type_id議運 / 本会議 / 会派代表者会議などの種別
sort_order同じ日の中での並び順
note補足
同日処理の基本ルール
  • 同じ日でも event_id が違えば別会議として扱います。
  • 同じ日の表示順は events.sort_order を使用します。
  • meeting_detail.html では、同じ日・同じ event_id の議題をまとめて表示します。

📋3. 議題一覧ページ（agenda.html）の仕様

議題マスタを基準にして一覧表示します。

3-1. 主な機能と表示列

  • 絞り込み： 大分類、中分類
  • フリーワード検索： 議題名、概要、備考、初回会議名
  • 表示列： 大分類、中分類、議題、初回日、最終日、関係会議数、履歴数

3-2. 集計ロジック

議題一覧では、agenda_actions を見て以下を自動計算します。

  • 初回日： 最も古い action_date
  • 最終日： 最も新しい action_date
  • 関係会議数： meeting_id のユニーク件数
  • 履歴数： agenda_actions の件数

🔍4. 議題詳細ページ（agenda_detail.html）の仕様

1つの議題に対して、いつ、どの会議で、どう扱われたかを時系列で表示します。

  • 基本情報： 大分類、中分類、内部ID、表示順、議題、概要、備考
  • 履歴表示（1行ごと）： 日時、会議回、会議種別、扱い（議題提示・協議等）、備考

🗓️5. 会議回詳細ページ（meeting_detail.html）での議題表示仕様

会議回詳細では、「会議回ベース」で議題を表示します。議題一覧や詳細が「議題ベース」なのに対し、こちらは「その回で何が扱われたか」に重点を置きます。

5-1. 表示ルール

  • 同じ日・同じ event_id は1つのグループにまとめます。
  • グループの上に「日時」「会議種別」を出します。
  • その下に、その会議で扱った議題（扱い内容を含む）を箇条書き風に並べます。
  • 議題の下の備考は、一覧性を高めるためここでは表示しません。

🛠️6. コード上の補助関数

各コード値を表示用の日本語ラベルに変換するための主要なマッピング関数です。

6-1. 大分類の変換（agendaCategoryLabel）

function agendaCategoryLabel(value) {
  const map = {
    STEERING: "議運",
    LEADERS: "会派代表者会議",
    OTHER: "その他"
  };
  return map[value] || value || "";
}

6-2. 中分類の変換（agendaSubcategoryLabel）

function agendaSubcategoryLabel(value) {
  const map = {
    SCHEDULE: "日程調整",
    BILL_HANDLING: "議案取扱",
    QUESTION_HANDLING: "一般質問取扱",
    COORDINATION: "調整",
    REPORT: "報告",
    OTHER: "その他"
  };
  return map[value] || value || "";
}

6-3. 扱いの変換（agendaActionTypeLabel）

function agendaActionTypeLabel(value) {
  const map = {
    PROPOSAL: "提示",
    DISCUSSION: "協議",
    DECISION: "決定",
    REPORT: "報告",
    CONTINUATION: "継続"
  };
  return map[value] || value || "";
}

🚀7. 今後の拡張について

必要に応じて、アジェンダに関する資料ファイル（PDF等）へのパスを持つ列を追加したり、特定の議題に関連する「議案」や「請願」と紐づける中間テーブル（アイテムとの紐付け）を作成することで、よりリッチな議会運営のトラッキングが可能になります。
