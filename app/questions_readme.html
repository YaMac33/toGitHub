questions系 説明書

開発者向け：「Excel列 → 出力JS → 画面での役割」整理

1. この仕組みの全体像

この questions 系は、Excelの2シートを元にして、VBAで2つのテキストJSへ書き出す構成です。

  • 通告ヘッダ シート → app/data/questions/questions.js → window.APP_DATA.questions
  • 通告明細 シート → app/data/questions/question_details.js → window.APP_DATA.question_details

VBAでは、シート名ごとに出力先ファイル名と window.APP_DATA のキー名を対応づけています。

2. 出力先の対応表

この対応はVBAの GetAllQuestionSheetConfigs() で定義されています。

Excelシート名 出力ファイル JS内の配列名
通告ヘッダ app/data/questions/questions.js window.APP_DATA.questions
通告明細 app/data/questions/question_details.js window.APP_DATA.question_details

3. 変換の基本ルール

VBAの BuildJsFromSheet は、Excelの1行目をそのままJSオブジェクトのキー名として使う作りです。
つまり、Excelの列名を変えると、そのまま出力JSのプロパティ名も変わります。

処理の流れ

  1. 1行目のヘッダ名を読む
  2. 2行目以降を1行ずつ読む
  3. 空行でなければ { key: value } 形式のJSオブジェクトにする
  4. それを配列として window.APP_DATA.questions などに格納する

4. 通告ヘッダシートの役割

通告ヘッダ は、「1人の議員の、1回の通告」につき1行持つ基本テーブルです。
ここには主に次のような情報を持たせます。

  • どの通告か
  • どの会議回か
  • いつの質問日か
  • 誰の通告か
  • 会派は何か
  • 通告番号はいくつか
  • 持ち時間は何分相当か

5. 通告明細シートの役割

通告明細 は、「その通告の中で、何を質問するか」を明細行で持つテーブルです。

ヘッダ = 通告そのもの
明細 = 通告の中身
※ question_id でヘッダと明細を結びます。

本来の考え方

今の設計意図としては、通告明細はできるだけシンプルにして、以下の列くらいに絞るのがきれいです。

detail_id, question_id, major_no, middle_no, minor_no, content, sort_order, created_at, updated_at

現在の実データ

今回貼ってもらった question_details.js には、上記以外にも次の列が残っています。

fiscal_year, meeting_id, number, year, session, notice_no, member_name, group
現在の実ファイルは「簡略化前の列も含んでいる状態」です。
ここは「今後整理する候補」として覚えておいてください。

6. 画面側ではどう使うか

画面（questions.html など）では、次の2ファイルを読み込んでいます。

  • ../../data/questions/questions.js
  • ../../data/questions/question_details.js

つまり画面側は、ヘッダ配列 questions と 明細配列 question_details の両方を使う前提です。実際の表示ロジックとしては、基本的にこう考えると整理しやすいです。

ヘッダで持つもの

  • 誰の通告か
  • 何回の定例会か
  • 通告番号
  • 質問日
  • 会派
  • 持ち時間

明細で持つもの

  • その通告の質問本文
  • 階層構造（大・中・小）

結合キー：question_id

7. VBAの型変換ルール

この説明はかなり重要です。将来「なんでこの値こう出るの？」となったときの原因になります。

数値として出力される列

VBAでは次の列名だけ数値扱いです。これらは文字列ではなく、数値としてJSに出力されます。

fiscal_year, year, notice_no, allotted_minutes, sort_order

日付として出力される列

  • question_date
  • 名前が _date で終わる列
  • 名前が _at で終わる列

これらは日付扱いです。さらに、
_at は yyyy-mm-dd hh:nn:ss
それ以外の日付は yyyy-mm-dd
で出力されます。

文字列として出力される列

上記以外は基本的に文字列として "..." で出力されます。そのため、
major_no, middle_no, minor_no は文字列のまま出力されます。VBAコメントにも「文字列出力版」と明記されています。

8. 空欄の扱い

VBAでは、次の値はすべて "" として出力されます。

  • エラー値
  • Empty
  • 空文字
  • 空白だけの値

つまり、空欄セルはJSでは基本的に空文字になります。
このため、たとえば sort_order を空欄にすると、出力JSでは "" ではなく数値変換対象なら場合によって "0" に寄る動きが気になることがあります。
特に数値列は NormalizeNumericLike の対象なので、運用上は 数値列はできるだけ空欄にしない 方が安全です。

9. 文字列エスケープのルール

文字列中の以下はVBA側でエスケープされます。

  • \ → \\
  • " → \"
  • 改行 → \n

つまり、Excelセル内に改行があっても、JSファイルとして壊れにくいように変換されています。

10. 開発者向けに覚えておくべきポイント

① Excelの列名を変えるとJSのキー名も変わる

このVBAは「列名固定の変換表」でなく、1行目のヘッダをそのまま使う方式です。
なので列名変更は画面JSにも影響します。

② 通告ヘッダと通告明細は question_id で結合する

ここが最重要です。
画面ロジックはこの前提で作るのが自然です。

③ 通告明細は今後さらに簡略化できる

今の実ファイルにはヘッダと重複する列が残っています。
ただ、設計としては「ヘッダに基本情報、明細に本文と階層だけ」という分離の方が分かりやすいです。

④ allotted_minutes はExcelの時間値由来の可能性が高い

サンプルでは 0.05 や 4.16666666666667E-02 のような値になっています。
これは見た目の分数・小数で保存されているので、画面表示時に「分」に変換する処理が必要です。保存値そのものは「表示用の分」ではない、と考えておいた方が安全です。

11. 一文サマリ

この questions 系は、
Excelの「通告ヘッダ」「通告明細」を、そのまま列名ベースでJS化して、question_id で結合して画面表示する仕組みです。

  • 基本情報は questions.js
  • 本文の階層構造は question_details.js
  • VBAは列名をそのままJSキーにする
  • 数値列・日付列だけ特別ルールで変換する
  • 将来的には question_details.js の重複列を減らせる
