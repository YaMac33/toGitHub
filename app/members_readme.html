members系 説明書

開発者向け：「Excel列 → 出力JS → 画面での役割」整理

1. この仕組みの全体像

members系は、Excelの複数シートをVBAでテキストJSに書き出し、それを members.html と member_detail.html で読み込んで表示する構成です。
VBAでは、各シート名ごとに 出力先ファイル と window.APP_DATA のキー名 が対応づけられています。

この系統は、単なる「議員一覧」ではなく、実際には次の複数データを束ねています。

  • 議員基本情報
  • 在任履歴
  • 会派履歴
  • 委員会履歴
  • 審議会履歴
  • 連絡先履歴
  • 特別委員会関連データ
  • 会議データとの関連
📌 結論： members系は「議員1人を中心に、各履歴テーブルを結合して表示する仕組み」です。

2. 出力先の対応表

VBAの GetAllMemberSheetConfigs() では、Excelシートと出力先が次のように定義されています。

Excelシート名 出力ファイル JS内の配列名
議員マスタapp/data/members/members.jswindow.APP_DATA.members
議員在任履歴app/data/members/office_terms.jswindow.APP_DATA.office_terms
会派マスタapp/data/members/parties.jswindow.APP_DATA.parties
議員_会派履歴app/data/members/member_parties.jswindow.APP_DATA.member_parties
委員会マスタapp/data/committees/committees.jswindow.APP_DATA.committees
議員_委員会所属履歴app/data/committees/member_committees.jswindow.APP_DATA.member_committees
審議会マスタapp/data/councils/councils.jswindow.APP_DATA.councils
議員_審議会所属履歴app/data/councils/member_councils.jswindow.APP_DATA.member_councils
議員連絡先履歴app/data/members/contacts.jswindow.APP_DATA.contacts
ここが重要で、以前の列整理よりも実際のVBAでは 「議員_審議会所属履歴」「議員連絡先履歴」も正式に出力対象 になっています。

3. 変換の基本ルール

questions系と同じく、members系のVBAも Excelの1行目のヘッダをそのままJSオブジェクトのキー名にする方式 です。つまり、Excelの列名を変えると、そのまま出力JSのプロパティ名も変わります。

  1. 1行目のヘッダを読む
  2. 2行目以降を1行ずつ読む
  3. 空行でなければ { key: value } の形にする
  4. それを window.APP_DATA.xxx = [...] として出力する

💡 この方式なので、列名の変更はVBA側ではなく画面JS側に効く、という意識が大切です。

4. 型変換ルール

members系VBAでは、列名によって値の型が変換されます。将来トラブル時に効く部分です。

4-1. 真偽値扱いの列

active_flag は特別扱いで、boolean風に正規化されます。ただしJSでは true / false ではなく、1 または 0 として出力されます。

  • "1", "true", "yes", "y" → 1
  • "0", "false", "no", "n" → 0
  • 数値なら 0 以外 → 1
  • それ以外 → 0

4-2. 数値扱いの列

次の列は数値として出力されます。

  • member_no
  • age
  • sort_order
  • elected_count

4-3. 日付扱いの列

列名が次に当てはまるものは日付として扱われます。

  • _date で終わる列
  • _at で終わる列
出力形式は members系では一律で yyyy-mm-dd です。
questions系のように _at を日時まで出す仕様ではありません。 ここは両者の違いです。

4-4. 文字列扱いの列

上記以外はすべて文字列として "..." で出力されます。空欄・Empty・エラー値は "" になります。

5. 文字列エスケープのルール

文字列中の以下はVBA側でエスケープされます。Excelセル内に改行や記号があっても、JSファイルとして壊れにくい設計になっています。

  • \ → \\
  • " → \"
  • 改行 → \n

6. 各シートの役割

members系は「議員1人に関する情報を分割保持している」ので、各シートの意味を未来の自分がすぐ思い出せるように整理します。

6-1. 議員マスタ → members.js

■ 役割

議員そのものの基本情報を持つ中心テーブルです。1議員につき1行を基本とし、他テーブルはここにぶら下がります。

■ 主な列

  • member_id
  • member_no
  • member_name
  • member_kana
  • member_name_short
  • birth_date
  • age
  • gender
  • sort_order
  • note
  • active_flag
  • created_at
  • updated_at

■ 画面での主な使われ方

member_detail.html では、基本情報欄に次の値が表示されます。
番号 / 在任状態 / 氏名かな / 性別 / 生年月日 / 年齢 / 備考

このうち member_no, member_kana, gender, birth_date, age, note は議員マスタ由来と考えてよいです。一方、在任状態 (current_status) は議員マスタ単独ではなく、履歴やAPP側の組み立て結果です。

■ 開発上の注意

member_id が議員の主キーです。一覧、詳細、履歴結合の中心になります。

6-2. 議員在任履歴 → office_terms.js

■ 役割

議員がいつからいつまで在任していたかを持つ履歴テーブルです。「現職かどうか」や「在任履歴一覧」の判定材料になります。

■ 主な列

  • office_term_id
  • member_id
  • term_start_date
  • term_end_date
  • election_label
  • elected_count
  • end_reason_code
  • note
  • created_at
  • updated_at

■ 画面での主な使われ方

member_detail.html の「在任履歴」で、各行は次のように組み立てられます。
在任期間 / 選挙名など (election_label) / 任期終了理由 (end_reason_code) / 備考 (note)

また、一覧画面の「現職のみ」フィルタや、詳細画面の current_status も、この履歴の情報をもとにAPP側で算出している可能性が高いです。

■ 結合キー

member_id

6-3. 会派マスタ → parties.js

■ 役割

会派そのものの名称マスタです。議員ごとの所属履歴とは分離されており、名称・略称などをここで持ちます。

■ 主な列

  • party_id
  • party_name
  • party_name_short
  • sort_order
  • active_flag
  • note
  • created_at
  • updated_at

■ 画面での主な使われ方

一覧画面では会派プルダウンの選択肢生成に使われます。view_members.js では window.APP.getPartyOptions() の結果を選択肢化しており、その元データには parties.js が使われています。
また、一覧表の「現在会派」表示や詳細の「現在会派」「会派履歴」表示にも使われます。

■ 結合キー

party_id

7. members.html で何をしているか

一覧画面 members.html は、単純な members.js だけでなく各履歴や特別委員会関係まで読み込んでいます。これは、一覧時点で「現在所属」や検索条件を組み立てる必要があるからです。

view_members.js では主に次を行っています。

  • 読み込み件数の表示
  • 会派/委員会選択肢の生成
  • 議員一覧の生成
  • フィルタ処理（現職のみ / 会派 / 現在委員会 / 氏名検索）
  • 行クリックで詳細画面を開く

つまり一覧画面は、議員マスタをベースにしつつ、現在所属情報を履歴から再構成した結果一覧です。

8. member_detail.html で何をしているか

詳細画面 member_detail.html は、URLの member_id を受け取り、その議員1人分の統合データを window.APP.buildMemberDetail(memberId) で組み立てて表示します。

逆算すると、APP側では少なくとも次のような統合オブジェクトを作っていると分かります。

  • 基本属性: member_id, member_name, member_no, age, current_status 等
  • 現在所属: current_party_name, current_committees_regular, current_councils 等
  • 履歴: party_history, committee_history_regular, office_terms 等

9. 特別委員会データとの関係

members系の画面では、通常の委員会データに加えて、次も読み込んでいます。

  • special_committees.js
  • special_committee_instances.js
  • special_committee_members.js
  • meetings.js

詳細画面の renderCommitteeHistory() では、常設委員会履歴と特別委員会履歴を分けて表示しています。
つまり、特別委員会は通常の member_committees.js とは別の構造で管理されており、会議回との関係まで含めて1つの履歴として表示していることになります。

10. 主要な結合キー

members系の肝は結合です。未来の自分向けには、ここを明記しておくのがかなり大事です。

中心キー つながる先 用途
member_id ほぼ全履歴 議員中心の統合（最重要）
party_id 会派マスタ 会派名取得
committee_id 委員会マスタ 委員会名取得
council_id 審議会マスタ 審議会名取得
特別委員会側のID群 特別委員会マスタ・履歴 特別委員会履歴取得

最重要は member_id です。members系は、議員1人を軸にすべてをぶら下げる構造です。

11. 開発上の注意点

11-1. 列名変更の影響が大きい

このVBAは列名をそのままJSキー名にするので、Excelの列名変更はAPP側ロジック破壊に直結します。

11-2. active_flag は 1/0 出力

真偽値っぽく見えても実際は 1 / 0 です。比較時に "1" なのか 1 なのかを混同しない方が安全です。

11-3. _at も日時ではなく日付出力

questions系と違い、members系では _at も yyyy-mm-dd です。作成日時・更新日時を時刻付きで保持したいなら、このVBA仕様を変える必要があります。

11-4. 画面表示の多くは「元データそのまま」ではなく「APPで統合後」

view_members.js と view_member_detail.js は、直接各配列を生で描画しているというより、window.APP.buildMemberList() や window.APP.buildMemberDetail() の結果を描画しています。
つまり、本当の表示仕様は APP層の組み立てロジックも見ないと完全確定しない です。

12. 一文サマリ

members系は、
「議員マスタを中心に、在任・会派・委員会・審議会・連絡先・特別委員会の各履歴を member_id などで結合し、一覧画面と詳細画面に再構成して表示する仕組み」です。

  • Excelの列名はそのままJSキーになる
  • VBAで複数の app/data/.../*.js を出力する
  • 一覧画面は「現在所属」を再構成して表示する
  • 詳細画面は「1議員ぶんの統合オブジェクト」を表示する
  • 特別委員会は通常委員会とは別系統で扱う
