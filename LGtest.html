Purpose

背景・目的

DX推進の検討材料

考えたこと

生成AI、特にLLMをうまく活用できれば、自治体業務でも 業務効率化につなげられる可能性がある。

実験の視点

  • AIを最大限活用するにはどうすればよいか
  • AI活用の可能性を広げるには何を知っておくべきか
  • LGWAN環境でも使える方法はあるか
本資料は、個人で試した内容を庁内共有向けに整理したものです。実運用には情報管理部門との調整が必要です。

Key Findings

今回の実験で分かった重要ポイント

結論
1

AIは便利だが揺れる

回答が毎回少し変わるため、正確性が必要な処理は仕組み化が必要。

2

オフラインでもできる

HTML・CSS・JavaScriptなら、LGWAN環境でも活用できる可能性がある。

3

属人化に注意

保守が必要なアプリを専門知識なしで作り込むのはリスクが高い。

Challenge

AIだけに業務処理を任せる難しさ

回答の揺れ

AIに任せきる場合

  • 同じ質問でも回答が変わる
  • 表現や判断に揺れが出る
  • 規定どおりの出力を保証しにくい

業務利用に向く形

  • 決められた情報を入力する
  • 決められた形式で出力する
  • 検索・閲覧はできるが、勝手に編集できない
AIは「最終処理」よりも、仕組みづくり・コード作成・資料作成の補助に使う方が現実的。

LGWAN Environment

LGWAN環境では「ほぼオフライン前提」で考える

制約の整理

LGWAN環境内ではインターネットへ直接接続できないため、半分オフラインのような感覚で考える必要があります。

  • ZIPファイルの直接ダウンロードは不可
  • 外部ライブラリの読み込みは難しい
  • ただし、HTMLファイルをブラウザで開くことは可能

Offline Possibility

オフラインでも意外とできることが多い

HTML活用

閲覧系

  • 業務マニュアル
  • 制度説明資料
  • 研修用スライド

検索系

  • 会議録検索
  • 通知文検索
  • FAQ検索

入力・出力系

  • 定型文生成
  • CSV出力
  • PDFへの書き込み補助
「インターネットにつながらない＝何もできない」ではなく、ブラウザで動く仕組みを作る余地があります。

Basic Terms

HTML・CSS・JavaScriptの役割

専門用語の補足

HTML

画面に表示する内容を作るもの。Webページの土台。

CSS

色、余白、文字サイズ、レイアウトなど見た目を整えるもの。

JavaScript

検索、表示切替、ボタン操作など画面上の動きを作るもの。

これらを1つのHTMLファイルにまとめれば、ブラウザでそのまま表示・利用できます。

Can / Cannot

できること・できないこと

整理
区分 できること 難しいこと
HTMLアプリ 閲覧、検索、画面上の入力、CSV・テキスト出力 Word・Excelファイルへの直接書き込み
データ保持 同じ端末・同じブラウザ内での一時保存 複数端末・複数職員での安定した共有
PDF 読み込み、画面上での書き込み、書き込み後の出力 高度な編集や複雑な帳票処理

Use Case

HTMLスライドとPowerPoint資料作成への応用

資料作成
  • HTMLで見栄えのよいスライドを作成できる
  • ただし、生成後に直感的に編集しにくい
  • そこで、同じ雰囲気のPowerPointを作るVBAコードをAIに生成させる
  • 最後はPowerPoint上で人が微調整する

Prompt Flow

AIを使った資料作成の流れ

手順
1
プロンプト生成

まずAIに、目的に合ったプロンプト文を作らせる。

2
HTML生成

生成したプロンプトでHTMLスライドのコードを作成。

3
VBA化

HTMLの雰囲気をPowerPointに変換するVBAを生成。

4
微調整

PowerPoint上で人が内容や配置を調整する。

いきなり完成物を依頼するより、プロンプト自体もAIに作らせると安定しやすい。

Development Support

VS Code・AIエージェントによるコーディング支援

開発補助

できたこと

  • HTMLベースの簡易Webアプリ開発
  • AIに質問しながらコード修正
  • GitHubへの反映作業

使った知識は限定的

  • git add .
  • git commit -m "メッセージ"
  • git push

高度な知識がなくても試せる一方で、コードを読めないまま進めることには限界があります。

GitHub

GitHubを使ったコード共有・公開

コード管理

GitHubとは

ソースコードを保存・共有・共同開発するためのWebサービス。 公開設定にすれば、ログインなしでコードを確認できる場合があります。

今回の使い方

  • VS Codeで作成したコードをGitHubへ送信
  • GitHub上の各ファイルからコードをコピー
  • LGWAN環境内で簡易アプリを再構築
手作業が多く、ファイル数が多いと煩雑。GitHubに置く情報の内容にも注意が必要です。

GAS Automation

Google Apps Scriptを使った自動化の可能性

GAS

Googleアカウントを利用できる環境があれば、Googleフォーム・スプレッドシート・GASを連携できます。

  • フォーム回答をスプレッドシートに自動反映
  • 行追加をきっかけにGASを実行
  • 通知や外部サービス連携も可能

Automation Flow

試した自動化の流れ

仕組み
1
Googleフォーム入力

公共PCなどからフォームに必要情報を入力。

2
スプレッドシート反映

回答内容が行として自動追加される。

3
GAS実行

行追加をトリガーにスクリプトが動作。

4
GitHub連携

認証情報を使ってファイル作成などを実行。

便利だが仕組みが複雑で、一般職員が通常業務の中で扱うには学習コストが高い。

Security

セキュリティ上の課題

最重要

リスクが高い運用

  • LGWAN内のテキストをネット側へ出す
  • 共有リンクを「全員編集可」にする
  • 個人アカウントで実験的に運用する

必要な対応

  • 情報管理部門との事前調整
  • 扱える情報範囲の明確化
  • 組織としての正式な運用ルール
実験としては可能でも、実運用では個人判断ではなく、組織的な確認と承認が必要です。

Risk

内製化で注意すべきこと

属人化
!
コードが読めないと保守が難しい AIに毎回全文を渡す必要があり、どこを直しているか把握しにくい。
!
継続的な運用が必要なアプリには不向き 担当者が異動した場合、仕組みが維持できなくなる可能性がある。
!
便利さとリスクはセット 作れることと、業務で安全に使えることは分けて考える必要がある。

Summary

まとめ

最終処理よりも、資料作成・コード作成・仕組みづくりの補助に向いている。

LGWAN内でも可能性あり
