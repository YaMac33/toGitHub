1このシステムについて

議員が一般質問の通告をWebフォームから提出し、その内容を事務局のスプレッドシートに自動集約するシステムです。提出と同時に、入力されたアドレスへ通告内容の全文を記載した控えメールが自動返信されます。

  • 議員が入力するのは メールアドレス・通告時間・質問内容 の3つだけ。発言順位・質問日・氏名・会派は名簿から自動で表示されます。
  • 質問内容は 大項目 → 中項目 → 内容 の階層で入力でき、番号は自動採番されます。
  • オンライン提出が難しい議員は、従来どおり 紙での提出 も併用できます（どちらでも運用は成立します）。
提出後の内容修正は 議会事務局を経由 する運用です（議員からの再提出・上書きはできません）。

2システム構成

Googleスプレッドシートに紐づいた1つの Apps Script ウェブアプリとして動作します。フォームの配信からデータ保存・控えメール送信まで、すべて事務局のGoogle環境内で完結します。

→

フォーム配信・名簿照合・検証・保存

ネットワークについて：スプレッドシートとGASはインターネット側のサービスのため、LGWAN端末からは利用できません。議員はインターネット系端末（SKY-DIV 等）からフォームを開く前提です。

5導入手順

  1. スプレッドシートを作成
    Googleドライブで新規スプレッドシートを作成（名称は任意。例「一般質問通告」）。中身は空のままでOK。
  2. スクリプトエディタを開く
    メニュー「拡張機能」→「Apps Script」。このシートに紐づいたスクリプトが開きます。
  3. 2つのファイルを貼り付け
    既定の コード.gs の中身を全消去して Code.gs を貼り付け。左の「＋」→「HTML」でファイル名 index を作成し、index.html を貼り付けて保存。
  4. シートを自動生成
    関数選択で setupSheets を選び「実行」。初回は権限承認を求められるので、事務局のアカウントで承認します。4シートが見出し付きで作成されます。
  5. 名簿・定例会名を入力
    「議員名簿」に各議員の情報を入力し、「設定」の定例会名を今回の名称に更新します。
  6. 公開（次節）
    ウェブアプリとしてデプロイし、発行URLを議員へ配布します。
権限承認の画面で「このアプリは確認されていません」と表示された場合：これは自作スクリプトのため通常表示されるものです。詳細を開き、自分のプロジェクトとして続行してください。

6公開（デプロイ）

  1. デプロイを開始
    エディタ右上「デプロイ」→「新しいデプロイ」→ 種類「ウェブアプリ」を選択。
  2. 実行ユーザーとアクセス範囲を設定
    「次のユーザーとして実行」＝自分（事務局アカウント）／「アクセスできるユーザー」＝全員 に設定。
  3. デプロイしてURLを配布
    発行されたウェブアプリURLを議員へ配布します。
なぜ「実行＝自分／アクセス＝全員」なのか：この組み合わせにより、議員はGoogleログイン不要でフォームを開け、名簿・集約データのスプレッドシートは事務局の権限で動くため議員側からは見えません。個人Gmailアカウントを使う場合でも成立する構成です。

7運用の流れ

  1. 議員がフォームを開き、メールアドレスを入力して照合。名簿に登録があれば氏名・会派・発言順位・質問日が自動表示されます（未登録ならその場で弾かれます）。
  2. 通告時間を選び、質問内容を入力。「プレビュー」で通告書の体裁を確認できます。
  3. 「提出する」で送信 → 提出マスタ・質問明細に記録され、入力アドレスへ控えメールが届きます。
  4. 事務局は質問明細の答弁者列を①単位で記入し、部局への割り振りや答弁者別一覧の作成に使います。
  5. 内容の修正依頼があれば、事務局がシート上で対応します（議員からの再提出はできません）。
一発提出：同一の定例会で同じアドレスからの二重提出は受け付けません（最初の1件のみ記録）。誤提出・修正は事務局へ。

8入力仕様・採番ルール

質問内容の階層と採番

階層呼称採番
第1階層大項目（件名）1. 2. 3. …
第2階層中項目（小見出し）(1) (2) (3) …
第3階層内容（質問文）① ② ③ …（⑳超は (21) 以降の表記で吸収）
  • 番号は自動採番。項目の追加・削除・並び替えに合わせて自動で振り直されます。
  • 構成は自由で、大項目のみ／大項目＋中項目／大項目＋中項目＋内容 のいずれでも提出できます（大項目の件名のみ必須）。

通告時間

30分〜80分・10分刻み（30/40/50/60/70/80分）から選択します。

出力

  • プレビュー（通告書の体裁・開閉式）
  • 印刷（A4縦レイアウト）。提出前は下書きとして印刷でき、提出後は完了画面から申請日時入りの正式版（控え）を印刷できます。
  • コピペ用テキスト（プレーンテキストでコピー）

9本人確認の考え方

本人確認は、入力されたメールアドレスを議員名簿と突き合わせる方式です。Googleアカウントのログイン認証は使いません。

  • 名簿に登録されたアドレスでなければ提出できません（未登録はその場で弾く）。
  • これは「登録済み議員のアドレスである」ことの確認であり、厳密な本人認証ではありません。アドレスを知っていれば他人を装える余地は理屈上残ります。
  • 抑止策として、提出時に控えメールが本人へ自動送信されます。身に覚えのない控えが届けば本人が気づけます。
  • 提出は数十名規模で、事務局が全件を確認する運用であること、紙提出も併存することから、実務上のなりすましリスクは低い前提です。
配布するウェブアプリURLは、必要な範囲（議員・関係者）にのみ共有してください。

10保守・更新

  • コードを修正したとき：「デプロイ」→「デプロイを管理」から既存デプロイを編集し、新しいバージョンを反映します（URLは変わりません）。「新しいデプロイ」を作り直すとURLが変わるので注意。
  • 定例会が変わったとき：「設定」シートの定例会名を書き換えます。
  • 改選・会派異動があったとき：「議員名簿」を更新します（メール・氏名・会派・発言順位・質問日）。発言順位・質問日が確定したら名簿に反映してから受付を開始します。
  • 過去の提出データ：提出マスタ・質問明細はそのまま蓄積されます。定例会名で絞り込めば回次ごとに区別できます。

11トラブルシューティング

症状確認・対処
フォームが真っ白／エラーになるHTMLファイル名が index になっているか確認。コード修正後に「デプロイを管理」で更新したか確認。
照合しても「登録されていません」になる名簿のメールアドレスに前後の空白や全角文字が混ざっていないか確認（半角で入力）。大文字・小文字と前後空白はプログラム側で吸収します。
控えメールが届かない迷惑メールフォルダを確認。個人Gmailアカウントは送信1日100通の上限があります（議員数なら通常問題ありません）。なお送信に失敗しても、提出データの保存自体は完了しています。
提出できない（二重提出の表示）同一定例会で既に提出済みです（仕様）。修正は事務局へ。
名簿に無い人が提出したい名簿に行を追加すれば提出可能になります。難しい場合は紙提出で対応。
権限承認の画面が出る自作スクリプトのため初回に表示されます。詳細を開いて自分のプロジェクトとして続行。
